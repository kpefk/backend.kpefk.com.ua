# CLAUDE.md — backend.kpefk.com.ua

## 1. Backend context

Цей backend розроблено для **КПЕФК** — українського закладу фахової передвищої освіти.

Backend розгортається на localhost і наразі є локальним середовищем розробки.

Backend **не є системою прийняття рішень**. Він зберігає та валідує адміністративно затверджені факти:

- педагогічні працівники (teachers)
- академічні групи (groups)
- студенти (students)
- навчальні плани (curricula)
- робочі навчальні плани (working curricula)
- призначення навчального навантаження (teaching load assignments)
- накази та підтвердження (orders and confirmations)
- користувачі та дозволи (users and permissions)

---

## 2. Законодавча база backend

Будь-яка бізнес-логіка backend, що стосується освітнього процесу, навчального навантаження, розподілу за підгрупами та підтвердження наказами, повинна перевірятись відносно цих джерел:

- **Закон України №2745-VIII «Про фахову передвищу освіту»**
  https://zakon.rada.gov.ua/laws/show/2745-19
- **Наказ МОН №686 від 18.06.2021** — норми часу для планування навчального навантаження
- **Наказ МОН №472 від 24.05.2022** — зміни до норм часу

### Ключові статті Закону №2745-VIII

| Стаття | Зміст |
|--------|-------|
| ст. 24 | Підвищення кваліфікації педагогічних працівників |
| ст. 32 | Структура закладу фахової передвищої освіти |
| ст. 37 | Педагогічна рада |
| ст. 47 | Форми здобуття освіти |
| ст. 49 | Навчальні плани — основа для структури `Curriculum` і `WorkingCurriculum` |
| ст. 59 | Посади педагогічних працівників |
| ст. 60 | Робочий час і навчальне навантаження |

### Критичні числові обмеження (ст. 60, Наказ №686)

- **720 годин** = навчальне навантаження на 1 ставку на рік
- Викладач може мати до **1.5 ставки**
- Отже backend повинен підтримувати ліміт до **1080 год/рік**
- **Не можна хардкодити припущення, що всі викладачі мають лише 1.0 ставку**

## 3. Backend principles

- Backend фіксує адміністративно затверджені факти, а не генерує їх самостійно.
- Якщо закон вимагає документ, протокол, наказ або підтвердження — backend зберігає метадані або статус цього документа.
- Не кожне юридичне правило має бути жорсткою помилкою валідації.

### Рівні валідації

| Рівень | Призначення |
|--------|-------------|
| **HARD BLOCK** | Неможливий стан даних, дублікати, порушені інваріанти, критичні доменні порушення |
| **SOFT WARN** | Рішення може існувати в реальності, але потребує уваги |
| **AUDIT INFO** | Юридично значущий факт, який повинен бути відстежуваний |

## 4. Backend architecture

**Стек:** NestJS · Prisma · PostgreSQL · конфігурація через env-файли для localhost

Backend має модульну структуру NestJS. Бізнес-логіка — лише в сервісах, не в контролерах.

### Модулі backend

| Модуль | Зміст |
|--------|-------|
| `auth / users` | Автентифікація, сесії, ролі, профіль |
| `teachers` | Педагогічні працівники, ставки |
| `groups` | Академічні групи |
| `students` | Студенти, прив'язка до групи |
| `curriculum` | Навчальні та робочі навчальні плани |
| `teacher-load` | Навантаження за планом |
| `teacher-load-assignments` | Фактичне призначене навантаження після затвердження |

### Центральні сутності бізнес-логіки

`WorkingCurriculum` · `CurriculumComponentTerm` · `TeacherLoadAssignment` · `Teacher` · `Group` · `User`

## 5. Core backend entities

| Сутність | Роль |
|----------|------|
| `Teacher` | Педагогічний працівник; ставка може бути до 1.5 |
| `Group` | Академічна група |

**`TeacherLoadAssignment` — центральна сутність backend для зберігання реально призначеного навчального навантаження після адміністративного затвердження.**

## 6. Backend domain rules

- `Teacher.rate` повинен підтримувати значення до 1.5.
- Ліміт навчального навантаження = `720 × Teacher.rate`.
- 1.5 ставки = 1080 год/рік.
- `subgroupNumber` є необов'язковим полем.
- Не можна припускати наявність підгрупи для кожного типу занять.
- Лекції, семінари, консультації, СПРС — не підгрупові за замовчуванням.
- `assignedById` і `signedByDirectorId` мають різний бізнес-сенс і не є взаємозамінними.
- `orderNumber` і `orderDate` — юридично значущі поля підтвердження наказом.
- Backend не повинен припускати, що вручну введені години є автоматично згенерованими.
- Backend повинен зберігати відстежуваність: хто призначив і хто підтвердив.

## 7. Backend coding rules for Claude Code

- Перед зміною бізнес-логіки backend — перевірити, чи стосується зміна правил із Закону №2745-VIII або Наказу №686.
- Не вводити довільні числові обмеження без правового або бізнесового обґрунтування.
- Не припускати, що кожен викладач має рівно 1.0 ставку.
- Не замінювати реальний адміністративний процес штучною автоматизацією.
- Дотримуватись суворої типізації TypeScript (strict-first, без `any`).
- Схема Prisma і валідація DTO повинні залишатись узгодженими.
- Зберігати сумісність із localhost-середовищем розробки.
- У коді: явні типи та зрозумілі назви англійською.
- Документація всередині CLAUDE.md — українською.

## Prisma rules

- **Prisma — єдиний доступ до БД.** Без сирих SQL-запитів поза Prisma.
- **Не видаляти поля** без явної причини.
- **Не змінювати `@map(...)`** без потреби — це змінює назви колонок і ламає міграції.
- **Усі зміни схеми — тільки через міграцію:** `npx prisma migrate dev --name <назва>`
- **Перевіряти `onDelete`** при додаванні зв'язків. Cascade може ненавмисно видалити пов'язані записи.

---

## Coding style

- **TypeScript strict-first.** `any` заборонений. DTO з `class-validator`.
- **NestJS pattern:** `Module → Controller → Service`. Бізнес-логіка — тільки в сервісі.
- **Logger:** `new Logger(ClassName.name)` — не `console.log`.
- **Коментар лише на "чому"**, не на "що".

---

## Anti-patterns

- Не додавати черги, кеші, event bus якщо задача вирішується простіше.
- Не генерувати архітектуру (CQRS, Event Sourcing, Repository pattern) без конкретної потреби.
- Не створювати абстракції наперед.
- Не припускати існування моделей, сервісів або полів — перевіряти в коді перед використанням.
- Не обходити `AuthGuard` / `RolesGuard` без явної причини в задачі.
