## 1. Backend context

Цей backend розроблено для **КПЕФК** (Ковельський промислово-економічний фаховий коледж ЛНТУ) — українського закладу фахової передвищої освіти.

Backend розгортається на localhost через Docker Compose (PostgreSQL + Redis) і наразі є середовищем активної розробки.

Backend **не є системою прийняття рішень**. Він зберігає та валідує адміністративно затверджені факти:

- педагогічні працівники (teachers) — синхронізуються з ЄДЕБО
- академічні групи (groups) — похідні від синхронізації студентів
- студенти (students) — синхронізуються з ЄДЕБО
- спеціальності та освітні програми (specialties, educational programs)
- навчальні плани та їх версії (curricula, curriculum versions)
- робочі навчальні плани (working curricula)
- призначення навчального навантаження (teacher load: subject + lesson assignments)
- вибіркові дисципліни (electives)
- аудиторний фонд (classrooms)
- накази та підтвердження (orders and confirmations)
- користувачі та дозволи (users and permissions)

- **720 годин** = навчальне навантаження на 1 ставку на рік
- Викладач може мати до **1.5 ставки**
- Отже backend повинен підтримувати ліміт до **1080 год/рік** (`720 × Teacher.rate`)
- **Не можна хардкодити припущення, що всі викладачі мають лише 1.0 ставку**
- Ліміт перевіряється **на етапі підтвердження (CONFIRMED)**, а не на етапі чернетки (DRAFT)

## 3. Backend principles

- Backend фіксує адміністративно затверджені факти, а не генерує їх самостійно.
- **Виняток:** `Student` і `Teacher` — це не "вручну введені факти", а дані, синхронізовані з ЄДЕБО (національний реєстр). Локальна адміністрація лише доповнює їх локально керованими полями (див. розділ 7 "EDBO синхронізація").
- Якщо закон вимагає документ, протокол, наказ або підтвердження — backend зберігає метадані або статус цього документа.
- Не кожне юридичне правило має бути жорсткою помилкою валідації.

### Рівні валідації

| Рівень | Призначення |
|--------|-------------|
| **HARD BLOCK** | Неможливий стан даних, дублікати, порушені інваріанти, критичні доменні порушення (напр. перевищення 720×rate) |
| **SOFT WARN** | Рішення може існувати в реальності, але потребує уваги (напр. дата наказу раніше 01.09) |
| **AUDIT INFO** | Юридично значущий факт, який повинен бути відстежуваний |

## 4. Backend architecture — неочевидні факти

Повний стек і структура `src/` — у @README.md. Тут лише те, що легко наплутати:

- Redis використовується як **сесійне сховище** (`connect-redis` + `express-session`), а **не** для JWT. Не пропонувати JWT-рішення для auth.
- Bun — основний package manager і runtime (`bun install`, `bunx prisma ...`). `package-lock.json` лишається для сумісності з Node.js, але не є основним шляхом.
- Локальний Docker Compose: PostgreSQL на порту **5433** (не 5432), Redis на **6379**.
- Бізнес-логіка — лише в сервісах NestJS, ніколи в контролерах.

### Центральні сутності бізнес-логіки

`Specialty` · `EducationalProgram` · `Curriculum` · `CurriculumVersion` · `WorkingCurriculum` · `CurriculumComponentTerm` · `TeacherLoadSubjectAssignment` · `TeacherLoadLessonAssignment` · `Teacher` · `Group` · `User`

Повний перелік сутностей — у @README.md ("Core domain entities"). Нижче — лише доменні факти, що **не випливають з назви чи з коду** і про які Claude легко помилиться:

## 6. User roles

Визначені в `prisma/schema.prisma` як enum `UserRole`:

| Роль | Опис | Що підтверджує/виконує |
|------|------|------|
| `STUDENT` | Студент | Перегляд власних даних, вибір вибіркових дисциплін |
| `TEACHER` | Викладач | Перегляд власного навантаження та кваліфікацій |
| `SCHEDULE_DISPATCHER` | Диспетчер розкладу | Керування розкладом |
| `HEAD_OF_DEPARTMENT` | Завідувач відділення | Нагляд за відділенням |
| `DEPUTY_DIRECTOR` | Заступник директора | **Генерує DRAFT навантаження**, керує навчальними планами; відповідає за `assignedById` |
| `DIRECTOR` | Директор | **Підтверджує навантаження наказом** (CONFIRMED), затверджує робочі плани; відповідає за `signedByDirectorId` |
| `ADMINISTRATOR` | Адміністратор | Повний доступ до системи |

> Звідси: `assignedById` (DEPUTY_DIRECTOR) і `signedByDirectorId` (DIRECTOR) мають різний бізнес-сенс і **не є взаємозамінними**.

## 7. EDBO синхронізація (ЄДЕБО) — критично для будь-якої логіки навколо Student/Teacher/Group

ЄДЕБО — національний реєстр освіти. Студенти й викладачі **не вводяться вручну** — вони синхронізуються щодня.

- **Що синхронізується:** `Student` (з `/api/studentEducations/list`), `Teacher` (з `/api/university/staff/list`), документи студентів (РНОКПП, паспорт, студентський — окремий backfill), освітні програми (з акредитаційними даними).
- **Де логіка:** `src/edbo/sync/edbo-sync.service.ts`, `src/edbo/sync/sync-state.service.ts`.
- **Розклад:** cron щодня о 02:00. Студенти, персонал і backfill документів виконуються паралельно; помилка в одному потоці не зупиняє інші.
- **Інкрементальність:** курсори зберігаються в таблиці `SyncState` (`students_last_sync_at`, `staff_last_sync_at`) і оновлюються **лише після успішного прогону**.
- **Overlap window:** 10 хвилин віднімається від останньої позначки часу — компенсація розсинхрону годинників між сервером і ЄДЕБО.
- **Ручний запуск** (тільки admin): `POST /edbo/sync/students`, `/staff`, `/all`, `/study-programs`.

> ⚠️ **HARD RULE:** sync використовує `upsert` і **навмисно не перезаписує локально керовані поля** (наприклад `Group.curatorId`). Будь-яка зміна sync-логіки повинна враховувати це. Перед зміною коду в `edbo/sync/` — обов'язково прочитати залежні файли: `SyncState`, `Student`, `Teacher`, `Group`, `StudentGroupHistory`.

> `Group` — похідна сутність: створюється автоматично під час синхронізації студентів, а не через окремий "create group" ендпоінт.

| Статус | Хто переводить | Умова переходу |
|--------|----------------|------------------|
| `DRAFT` | `DEPUTY_DIRECTOR` | Генерується з `WorkingCurriculum` + `CurriculumComponentTerm` |

| `CONFIRMED` | `DIRECTOR` | Обов'язкові `orderNumber` + `orderDate`; HARD BLOCK при перевищенні 720×rate; SOFT WARN якщо дата наказу раніше 01.09 |

## 9. Backend domain rules

- `Teacher.rate` повинен підтримувати значення до 1.5.
- Ліміт навчального навантаження = `720 × Teacher.rate`, перевіряється **на етапі CONFIRMED**.
- `subgroupNumber` є необов'язковим полем.
- Не можна припускати наявність підгрупи для кожного типу занять. Лекції, семінари, консультації, СПРС — не підгрупові за замовчуванням.
- `assignedById` і `signedByDirectorId` мають різний бізнес-сенс і не є взаємозамінними (див. розділ 6).
- `orderNumber` і `orderDate` — юридично значущі поля підтвердження наказом.
- Backend не повинен припускати, що вручну введені години є автоматично згенерованими.
- Backend повинен зберігати відстежуваність: хто призначив і хто підтвердив.
- **`CurriculumVersion` незмінна після публікації.** Виправлення помилок — лише через створення нової версії, а не редагування опублікованої.
- **Локально керовані поля sync-сутностей (`Group.curatorId` тощо) захищені від перезапису** при синхронізації з ЄДЕБО.

## 10. Backend coding rules for Claude Code

- Перед зміною бізнес-логіки backend — перевірити, чи стосується зміна правил із Закону №2745-VIII або Наказу №686 (з урахуванням примітки в розділі 2 про актуальність редакції).
- Не вводити довільні числові обмеження без правового або бізнесового обґрунтування.
- Не припускати, що кожен викладач має рівно 1.0 ставку.
- Не замінювати реальний адміністративний процес штучною автоматизацією.
- Не плутати синхронізовані сутності (`Student`, `Teacher`) із суто адміністративними (`WorkingCurriculum`, `TeacherLoadSubjectAssignment`) — у перших є upsert-захист локальних полів, у других немає.
- Перед зміною sync-логіки — прочитати всі залежні файли (`SyncState`, `Student`, `Teacher`, `Group`, `StudentGroupHistory`).
- Дотримуватись суворої типізації TypeScript (strict-first, без `any`).
- Схема Prisma і валідація DTO повинні залишатись узгодженими.
- Зберігати сумісність із localhost Docker Compose середовищем (Postgres :5433, Redis :6379).
- Не обходити сесійну автентифікацію через Redis власною JWT-логікою.
- У коді: явні типи та зрозумілі назви англійською.
- Документація всередині CLAUDE.md — українською.

## Prisma rules

- **Prisma — єдиний доступ до БД.** Без сирих SQL-запитів поза Prisma.
- **Не видаляти поля** без явної причини.
- **Не змінювати `@map(...)`** без потреби — це змінює назви колонок і ламає міграції.
- **Усі зміни схеми — тільки через міграцію:** `bunx prisma migrate dev --name <назва>` (локально), `bunx prisma migrate deploy` (production/CI).
- **Ніколи не редагувати `schema.prisma` без подальшого запуску `migrate dev` локально.**
- **Перевіряти `onDelete`** при додаванні зв'язків. Cascade може ненавмисно видалити пов'язані записи — особливо обережно з EDBO-синхронізованими сутностями.

---

## Coding style

- **TypeScript strict-first.** `any` заборонений. DTO з `class-validator` + `class-transformer`.
- **NestJS pattern:** `Module → Controller → Service`. Бізнес-логіка — тільки в сервісі.
- **Logger:** `new Logger(ClassName.name)` — не `console.log`.
- **Коментар лише на "чому"**, не на "що".
- **Пакетний менеджер:** Bun (`bun install`, `bun run ...`). `package-lock.json` присутній лише для сумісності з Node.js — не використовувати `npm install` як основний шлях.

---

## Anti-patterns

- Не додавати черги, кеші, event bus якщо задача вирішується простіше.
- Не генерувати архітектуру (CQRS, Event Sourcing, Repository pattern) без конкретної потреби.
- Не створювати абстракції наперед.
- Не припускати існування моделей, сервісів або полів — перевіряти в коді перед використанням.
- Не обходити `AuthGuard` / `RolesGuard` без явної причини в задачі.
- Не вважати `Student`/`Teacher` "вручну введеними" сутностями — це sync-дані з захищеними локальними полями.
- Не робити великі рефакторинги в одному коміті, що змішують feature-роботу з реструктуризацією.
